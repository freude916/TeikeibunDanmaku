# Timepoint 编写说明本文整理本项目里 `timepoint` 的固定写法，目标是让你新增一个事件点时只做最小改动、快速可用。请注意这是描述性文档，除非进行了重构改动，否则不要改动这篇文档增加新的 timepoint 相关说明。新增 timepoint 时只需要按照这个文档的步骤操作即可。timepoint 和 state 的描述列表在 docs/timepoint_list.md，新增 timepoint 时请务必补充。## 1. Timepoint 是什么- `timepoint` 是规则触发的事件 ID（例如 `reward.seen`、`shop.seen`）。- 每次事件发生时，会携带一个 `BoardState`（当前上下文数据）并通过 `TimepointBus` 发布。- `RuleEngine` 订阅总线，按 `timepoint + condition` 匹配规则并发送弹幕。关键链路：`Detection (Harmony Patch)` -> `Timepoint.Publish()` -> `RuleEngine.OnTimepointPublished()`## 2. 固定结构### 2.1 基类约定见 [Timepoint.cs](/home/zed/Documents/RiderProjects/TeikeibunDanmaku/TeikeibunDanmakuCode/Timepoints/Timepoint.cs)：- `Timepoint.Id`：字符串事件 ID。- `Timepoint.BoardState`：规则条件读取的数据。- `Timepoint<TState>.State`：强类型状态。- `Publish()`：发布到 `TimepointBus`。### 2.2 一个具体 Timepoint 的最小模板```csharppublic sealed class XxxTimepoint : Timepoint<CardState>{    public const string TimepointId = "xxx.yyy";    public const string TimepointDisplayName = "中文显示名";    public override string Id => TimepointId;    public override string DisplayName => TimepointDisplayName;    public static XxxTimepoint From(CardModel model)    {        ArgumentNullException.ThrowIfNull(model);        return new XxxTimepoint { State = CardState.FromCardModel(model) };    }}```可参考：- [RewardSeenTimepoint.cs](/home/zed/Documents/RiderProjects/TeikeibunDanmaku/TeikeibunDanmakuCode/Timepoints/RewardSeenTimepoint.cs)- `CardState`）。- 不要跨层塞不该有的数据；Detection 只负责采集和发布。### 步骤 2：新增 `Timepoints/*Timepoint.cs`- 定义 `TimepointId` 常量（统一小写点分，如 `shop.seen`）。- `Id` 返回该常量。- 提供 `From(...)` 工厂方法，内部做 `null` 检查并构造 `State`。### 步骤 3：在对应 Detection 发布- 在 Harmony Patch 的 Postfix/Prefix 中提取数据。- 遍历业务对象，逐条调用 `XxxTimepoint.From(xxx).Publish()`。- 只做事件采集，不要在 Detection 里写规则逻辑。参考：- [CardRewardOpenedDetection.cs](/home/zed/Documents/RiderProjects/TeikeibunDanmaku/TeikeibunDanmakuCode/Detection/Rewards/CardRewardOpenedDetection.cs)- [ShopOpenedDetection.cs](/home/zed/Documents/RiderProjects/TeikeibunDanmaku/TeikeibunDanmakuCode/Detection/Shop/ShopOpenedDetection.cs)### 步骤 4：注册状态解析在 [TimepointStateResolver.cs](/home/zed/Documents/RiderProjects/TeikeibunDanmaku/TeikeibunDanmakuCode/Timepoints/TimepointStateResolver.cs) 里加映射：```csharpnew(ShopSeenTimepoint.TimepointId, ShopSeenTimepoint.TimepointDisplayName, typeof(CardState))```不注册会导致规则反序列化时报 `Unknown timepoint id`。同时这一步也决定 RuleEditor 下拉里显示的时间点名，所以 `TimepointDisplayName` 和 resolver 描述符都必须补。### 步骤 5：检查 State 的字段显示名RuleEditor 的字段展示名来自 `BoardState` 上的 `[DataField("中文名")]`。- 如果你复用已有 `CardState`：确认已有字段名够用。- 如果你新增 `XxxState`：所有参与规则匹配的属性都要加 `[DataField("中文名")]`，否则 UI 会退回英文属性名，后续也容易漏维护。## 4. 规则文件怎么写规则通过 `timepoint` 字段绑定事件：```json{  "rule_id": "example_shop_seen",  "timepoint": "shop.seen",  "condition": {    "type": "eq",    "key": "ModelId",    "value": "PERFECT_STRIKE"  },  "messages": [    "商店也刷这个？"  ]}```其中 `key` 来自对应 `State` 上的 `[DataField]` 字段（例如 `CardState.ModelId`）。`combat.card_played` 的示例规则：```json{  "rule_id": "example_combat_card_played",  "timepoint": "combat.card_played",  "condition": {    "type": "gt",    "key": "DamageTotal",    "value": "15"  },  "messages": [    "这张打得真狠"  ]}```## 5. 自检清单- 新 `timepoint` 类已创建，`Id` 与常量一致。- 新 `timepoint` 已提供 `DisplayName`（中文可读名）。- Detection 已调用 `Publish()`，且有必要的空值保护。- `TimepointStateResolver` 已注册 `TimepointDescriptor(Id, DisplayName, StateType)`。- 规则 `timepoint` 字符串与常量完全一致。- 若新增了 State/context，规则字段都带 `[DataField("中文名")]`。- 构建检查优先执行 `./dotcheck.sh`。# 请再次注意：这是描述性文档，除非进行了重构改动，否则不要改动这篇文档增加新的 timepoint 相关说明。新增 timepoint 时只需要按照这个文档的步骤操作即可。**timepoint 和 state 的描述列表在
