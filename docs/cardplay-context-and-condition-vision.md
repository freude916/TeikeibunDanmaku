# CardPlay / Context / Condition 重构愿景（讨论稿）## 1. 背景当前项目已经有 `CardPlayedTimepoint` 与 `CardPlayState`，可以覆盖单次出牌的即时信息（例如伤害、费用、series）。但业务需求已经出现“即时 + 长期”混合判断：- 例：牌组里“毒”很多，并且当前遇到/打出了“毒”的终端牌 -> 发送特定弹幕。这类规则如果每次 timepoint 都全量扫描 deck，会产生重复计算与结构耦合问题。## 2. 目标- timepoint 仍然只负责“事件触发 + 本次快照”。- 长期上下文使用缓存（按 run/combat 生命周期维护），避免每次触发都重算。- condition 结构能表达集合/映射语义，而不是把所有信息塞进一个字符串。- RuleEditor 按 condition 类型渲染专用输入控件，避免让用户背“编码格式”。## 3. 上下文分层（建议）### 3.1 短期上下文（Ephemeral）由当前 timepoint 直接采集，生命周期只覆盖本次事件。- 例如：`CardPlayState`（本次出牌伤害、是否末段等）### 3.2 长期上下文（Cached）由专门 collector 维护，在事件发生时读取增量，规则评估时直接取值。建议先做两层：- `RunContextCache`：跨战斗持续（牌组、遗物、全局计数）- `CombatContextCache`：单场战斗持续（本场累计伤害、打牌数、回合统计）### 3.3 聚合读取（Resolver）规则评估时，timepoint state 与 cache state 通过统一入口暴露给 condition：- 不要求 UI/规则作者关心数据来自哪个层。- 但代码层仍保持分层，不跨层写逻辑。## 4. 针对“毒很多 + 毒终端”的字段设计不要把流派判断仅做成自由文本；建议同时保留严格字段。### 4.1 Card（单卡）- `IsTerminal: bool`- `Archetypes: IReadOnlyList<string>`（允许一张卡多个流派）### 4.2 Deck（缓存）- `ArchetypeCounts: IReadOnlyDictionary<string, int>`这样可以自然表达：- `eq(IsTerminal, true)`- `list_contains(Archetypes, "毒")`- `dict_count_gte(ArchetypeCounts, entry="毒", threshold=6)`## 5. Condition 结构升级（建议）当前 `value` 是单值输入，表达能力不足。建议升级为“按类型定义参数结构”。### 5.1 新条件类型（第一批）- `list_contains`：列表包含某项- `list_contains_any`：列表包含任一项- `dict_count_gte`：字典指定 key 的计数 >= 阈值### 5.2 DTO 形态保留 `type/key`，`value` 改为结构化对象（JSON 对象）：- `list_contains`: `{ "item": "毒" }`- `dict_count_gte`: `{ "entry": "毒", "threshold": 6 }`## 6. RuleEditor 结构升级（建议）从“统一 value 文本框”改为“按 condition 类型渲染参数面板”。- `eq/lt/gt/find`：保留单值控件- `list_contains`：单字符串输入（后续可替换成枚举/tag 选择）- `dict_count_gte`：`entry` 文本框 + `threshold` 数字框编辑器职责：- 根据 `type` 决定参数 UI- 根据 `key` 的字段类型约束可选 condition- 在 UI 层即时校验参数完整性与类型## 7. 迁移策略（分期）### Phase A（最小可用）- 加 `RunContextCache` 的 deck 流派计数- 新增 `list_contains` 与 `dict_count_gte`- RuleEditor 支持这两个条件的参数控件### Phase B（扩展）- 补 `list_contains_any` 等高频集合条件- 字段元数据补“候选值提示”（可选）### Phase C（清理）- 清理旧规则里靠字符串拼接实现集合语义的写法- 补回归测试：解析、校验、评估、编辑器交互## 8. 风险与边界- 不做脚本型 condition（如内嵌表达式执行），避免失控。- 不在 Detection 里做复杂聚合，聚合都进入 cache collector。- 不把 UI 与 evaluator 强绑定在具体字段名上，靠 schema 描述驱动。## 9. 待确认事项- 流派 token 规范：是否统一小写英文 token（如 `poison`）并在 UI 显示中文？- `Archetypes` 的来源：仅卡牌 catalog，还是允许关键词推断补充？- 阈值默认值：`dict_count_gte` 初始阈值是 `1` 还是 `0`。
